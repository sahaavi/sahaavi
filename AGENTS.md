# GitHub Profile README Instructions

This repository is the special GitHub profile repository for `sahaavi`.
Its `README.md` appears on the public GitHub profile.

When updating `README.md`, follow this style contract:

- Blend the GitHub profile README styles of:
  - Chip Huyen: human, first-person, memorable, technically credible.
  - Eugene Yan: current-work oriented, with writing, learning, and practical applied ML themes.
  - Sebastian Raschka: clean, short, expertise-led, with links doing most of the credibility work.
- Keep the README current, personal, understated, and proof-led.
- Lead with what Avishek is exploring or building now, not a job title.
- Let links and pinned repositories carry most of the proof.
- Prefer concrete current work over broad skill lists.
- Mark unfinished projects clearly as in progress.
- Make the page feel like a technical working notebook and project index, not a public resume.

Avoid:

- Resume-like sections such as `Background`, `Education`, `Core Stack`, `Work Experience`, `Flagship Project`, `Current Build`, or `Earlier Proof` unless explicitly requested.
- Employer details, education details, and long technology inventories unless explicitly requested.
- Badge walls, stats cards, generic AI buzzwords, and sales-copy phrasing.
- Direct email exposure unless explicitly requested.
- Overclaiming that a demo is production software or that an in-progress RAG system has completed evaluation.
- Reusing these rejected framings: "camera input, speech, retrieval, tools, confirmations, and backend pieces" or "boring-but-important parts."

Preferred README shape:

1. H1 with the name.
2. Two short first-person paragraphs about current technical interests.
3. A `Now` section with 2-3 concise bullets and compact links.
4. Optional `Notes` or `Writing` section if there is real public work to link.
5. Minimal `Elsewhere` section with portfolio and LinkedIn.

Before changing the README in future sessions, inspect current public project state and draft first if the user asks to review before applying. After changing it, verify Markdown rendering and links before committing.
